Linters

A Linter is a tool within static code analysis that examines source code to flag mistakes, anomalous code, probable bugs and stylistic errors.

But it can be customized to fit your teams code standards also, everything from spelling mistakes to formatting, really depends on the teams use case.

They help analyze the codebase before you execute it, which can help in the early stages of development and beyond.

I personally use Prettier but there are tons of linters like ESlint, JShint, StandardJS and etc.


Git in the terminal

Git is a free and open source distributed version control system

We've mainly used Git to push code to our repositories on Github but there are other sites where you can store code like Azure Dev

The usual terminal commands i use are:

git add -A

Which stages all changes


After that i commit my code with:

git commit -m "message"

Where i commit my code with a message appropriate for the commit i have made.


And at last i push my code: 

git push origin main

I usually never push directly to main, i often use branches with:

git checkout -b "branchname"

And handle merging by creating pull requests on Github


Data types in JS

Here i will go over the different data types we have gone trough for now

Number

The number datatype can contain whole numbers, decimal numbers and also NaN

For example:

const count = 10;
const price = 14.50;
const notANumber = NaN;

String

The String datatype is a sequence of characters

Examples of strings:

const fruit = "apple";
const name = 'Jonas';        
const lastName = ´Nilket-Johannesen´;

Boolean

The boolean datatype can be either true or false

const isOnline = true;
const isLoggedOff = false;

Undefined

The Undefined datatype is a varible with no assigned value:

let x;

Null

The Null datatype is a value representing object absence

let x = null;

Object

The Object datatype is a collection of key-value pairs of data:

const peron = {firstname: "Jonas", lastName: "Nilket-Johannesen"}

There is also the Array Object which is a array of values accessed by a numerical index:

const cars = ["Volvo", "Toyota", "Koenigsegg"]

Function

A function in Javascript is also a datatype. A function is a reusable block of code, that can be called.

Functions can be passed as an argument in other functions also called a callback function.

Example of a simple function:

function cat() {
    return "meow";    
}

Type coercion

Javascript has built-in type coercion that happens automatically

Some say it's a feature others hate on it.

And it's not always predictable from a logical sense


Examples:

'10' + 7; = "107"
'10' - 7; = 3
false + 1; = 1
null + 10; = 10
undefined - 27; = NaN

Type coercion

In type coercion there is also falsy values like:

False, null, undefined, 0, NaN and ""

When you try to use one of these values in a boolean expression, Javascript will automatically coerce it

To a boolean value, for example:

if(0) {
    //This code will not execute
} else {
    //this code will execute
}

== vs ===

When you compare data in Javascript it's a big deal whether you choose to use == or ===

Using == will trigger type coercion, which can allow for some unpredictable behaviour, and when you use === it will not trigger type coercion which means it won't convert the datatype.

Examples:

0 == false; = true <- type coercion
"" == false; = true <- type coercion
0 === false; = false <- No coercion
null === undefined: = false <- No coercion

REST API

Rest is an architectural style, that we follow while buuiding web API's

HTTP methods

There are five main HTTP methods that we have gone over through this semester and they are:

1. GET - Represents methods that gather data

2. POST - Represents methods that creates new data

3. PUT - Represents methods updates a whole row/object

4. PATCH - Represents methods that update parts of a row/object

5. DELETE - Represents methods that deletes data

Standards and conventions

Many people tend to add the /api prefix to their endpoints, to seperate rest api endpoints and frontend site serving endpoints

At the same time it's also mandatory to have whatever object your calling your endpoints in plural inside your url, an example of both is:

/api/beers

The order of the endpoints

Is also an important subject while working with rest, it should follow this order everytime:

GET /users -> get all users
GET /users/{id} -> get user by id
POST /users -> create a new user
PUT/ /users/{id} -> update user by id
PATCH /users/{id} -> update specific fields for a user by id
DELETE /users/{id} -> delete a user by id
